# Care docs conventions (shared)

These apply to every concept and reference page. The build (`npm run build`) enforces most of them, so getting them right means a green CI.

## Repository layout

- Docs site: this repo (`code/docs`), a Docusaurus site.
- Care backend: a sibling checkout (`../care` or `code/care`) — the source of truth for all technical content.
- Versions: `versioned_docs/version-3.0/` and `versioned_docs/version-3.1/` are kept **byte-identical**. Author in `3.0`, then mirror to `3.1`.
- Locales: English + Malayalam (`ml`). Only the patient docs are translated under `i18n/ml/`; everything else falls back to English.

## Domains & slugs

Docs are organized into domain folders, each with a `_category_.json` (label, position, generated-index). The sidebar is `autogenerated`, so a new file appears automatically — no sidebar editing needed. Use the existing domain that fits; create a new domain folder + `_category_.json` only for a genuinely new area.

**Reference domains** (`references/<domain>/`): `foundation`, `clinical`, `definitions`, `medications`, `scheduling`, `billing`, `supply`, `facility`, `access-governance`, `platform`.

**Concept domains** (`concepts/<domain>/`): the same, minus `foundation` (base models aren't a product concept).

Grouping logic:
- `clinical` — the patient record and clinical data, including `questionnaire-response`.
- `definitions` — reusable definitions, templates, and terminology: `activity-definition`, `observation-definition`, `specimen-definition`, `charge-item-definition`, `product-knowledge`, `questionnaire`, `questionnaire-response-template`, `valueset`. A new resource that *defines* structure for instance records goes here.
- `access-governance` — the instance-level access model: `organization`, `facility-organization`, `user`, and (references) `role`/`permission`/`permission-association` or (concepts) `access-control`.
- `medications`, `scheduling`, `billing`, `supply`, `facility` — operational domains; `platform` — cross-cutting plumbing.

Slugs are kebab-case and match between a concept and its reference where both exist (e.g. `clinical/encounter`, `billing/charge-item`). One primitive per page (de-clustered) — never bundle several resources into one page.

## Links (critical — the build runs all locales with `onBrokenLinks: 'throw'`)

- **Normal internal links use a relative path WITH `.mdx`**: a sibling `./<slug>.mdx`, a cross-domain `../<domain>/<slug>.mdx`, a concept↔reference `../../references/<domain>/<slug>.mdx`. `.mdx` links are file-relative and robust (they survive trailing-slash and `X/X` folder-name collisions).
- **Extensionless links ONLY for the translated docs** — `patient` and `create-patient`. Because only those are translated in `ml`, a `.mdx` link to them breaks the Malayalam build (Docusaurus resolves `.mdx` by source-file path, and the translated file lives in a different root). So link them as `../clinical/patient`, `../../concepts/clinical/patient`, `../../flows/clinical/create-patient` — no extension.
- Never invent a path. If unsure a target exists, link the GitHub source instead.

## MDX safety (these silently break the build)

- **No `{#custom-anchor}` heading ids.** In `.mdx`, `{#...}` is parsed as a JSX expression and fails (`#` is invalid JS). Rely on Docusaurus's auto-generated heading ids; if you link to a heading, make the link target match the slugified heading text.
- **Wrap any literal `{ }` in backticks or a fenced code block.** A bare `{` in prose/tables is parsed as a JSX expression. JSON-shape blocks belong in ```` ```text ```` fences.
- **Never write tool-call XML** (`</content>`, `</invoke>`, `antml:` …) into a file. If you see such a tail (agent leakage), strip it.

## Mirror 3.0 → 3.1

After writing/updating `version-3.0`, copy the changed files to `version-3.1` so the two stay identical. For a few files, `cp` each one; for a full rebuild of a folder, `rm -rf versioned_docs/version-3.1/<folder> && cp -R versioned_docs/version-3.0/<folder> versioned_docs/version-3.1/<folder>`. The `_category_.json` files mirror too.

## Build validation (the authoritative gate)

Always run the **full** build before declaring done — CI builds every locale:

```bash
npm run build
```

Look for `[SUCCESS] Generated static files in "build"` AND `"build/ml"`, exit code 0. Do **not** rely on `--locale en` alone — it hides the Malayalam i18n breakages. The build catches broken links, broken heading anchors, MDX parse errors, and frontmatter problems. Fix anything it reports and rebuild until clean.

A quick pre-build corruption scan is cheap insurance:

```bash
grep -rlE '</?(content|invoke|parameter)\b|antml:|\{#' versioned_docs --include='*.mdx'
```
