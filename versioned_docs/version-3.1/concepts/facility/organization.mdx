# Organization

An **organization** is a node in Care's governance tree — a way to group people and grant them access without tying that access to a single facility. It is the backbone of who-can-do-what across the platform: a role granted on a parent organization flows down to everything beneath it.

## What it represents

In Care's FHIR-aligned model, an organization maps to the **Organization** resource. Each one carries a name and description, a type that says what kind of grouping it is, a position in a nested hierarchy, and a set of members who each hold a role.

An organization is not a facility. A facility is a physical place where care happens; an organization is an administrative grouping that exists instance-wide, used to organize permissions and reach. The departments and teams *inside* one hospital are modelled separately as [facility organizations](./facility-organization.mdx) — this concept is the platform-wide tree above them.

## Types

The type tells Care how a node behaves and who is allowed to manage it.

| Type | What it is |
| --- | --- |
| `team` | A team or department-style grouping of people |
| `govt` | A governance or governmental unit, typically managed by superadmins |
| `role` | A role-grouping organization, used to assign a shared role to a set of users |
| `product_supplier` | A supplier organization that links into the supply chain |

Most everyday groupings are `team` organizations. Governance and role organizations, and any root of the tree, are restricted to superadmin management because they shape access for everyone below them.

## How the tree works

Organizations nest. A root sits at the top with no parent, and every other node points to a single parent, forming a chain like `Government → State Health Dept → District → Cardiology team`.

The defining behaviour is **inheritance**: a role granted on a parent applies implicitly to all of its descendants. Give someone an admin role on a district and they can act across every team in that district without being added to each one. This is what makes the tree a permission structure and not just a folder system — a permission held high in the tree reaches everything below it.

A few rules keep the tree consistent: siblings under the same root must have unique names; a parent cannot be deleted while it still has children; and a node's parent is fixed once created — you move people, not the node.

## Membership

People gain access by being added to an organization with a **role** — that pairing of a user and an organization is what grants permission. Adding someone to a district as an admin is very different from adding them as a member, even though both are "members of the district."

Membership grants access *through* the organization, not *to* everything it touches. A user can act on the resources assigned to their organization and, by inheritance, those of its descendants — always bounded by what their role permits.

## Permissions

Access to organizations is governed by the permissions below. Each lists the system roles that carry it by default.

| Permission | Description | System Roles |
| --- | --- | --- |
| `can_create_organization` | Create a new organization under a parent the user can already manage (root, governance, and role organizations remain superadmin-only) | Admin |
| `can_view_organization` | View organizations and list/retrieve them through the accessible-organizations filter | Facility Admin, Admin, Staff, Doctor, Administrator, Nurse, Volunteer, Pharmacist, Admin (role org), Manager (role org), Member (role org) |
| `can_manage_organization` | Update an organization's name, description, and metadata, manage its connected role organizations, and delete it once it has no children | Admin, Admin (role org) |
| `can_list_organization_users` | List the members of an organization | Facility Admin, Admin, Staff, Doctor, Administrator, Nurse, Volunteer, Pharmacist, Admin (role org), Manager (role org) |
| `can_manage_organization_users` | Add, remove, and assign roles to members of an organization (bounded by the actor's own role privileges) | Admin, Administrator, Facility Admin, Admin (role org) |
| `can_manage_connected_role_organizations` | Add, remove, and assign roles to members of connected role organizations | Admin (role org), Manager (role org) |
